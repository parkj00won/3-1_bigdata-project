## 1. Pandas 핵심 자료구조 복습

### 1-1. Series와 DataFrame

**NumPy란?**

NumPy는 Python에서 **수치 계산(숫자 계산)**을 빠르고 효율적으로 하기 위한 핵심 라이브러리이다. Python의 기본 리스트(list)로도 숫자를 다룰 수 있지만, 데이터가 수만~수백만 건이 되면 속도가 매우 느려진다. NumPy는 C언어로 내부 구현되어 있어 대량의 숫자 데이터를 빠르게 처리할 수 있다.

```python
import numpy as np

# Python 리스트 vs NumPy 배열
python_list = [1, 2, 3, 4, 5]
numpy_array = np.array([1, 2, 3, 4, 5])

# NumPy는 배열 전체에 연산을 한 번에 적용할 수 있다 (벡터 연산)
print(numpy_array * 2)      # [2, 4, 6, 8, 10] — 반복문 없이 한 번에 처리
print(numpy_array.mean())    # 3.0 — 평균 계산
print(numpy_array.std())     # 1.414... — 표준편차 계산
```

> Pandas의 Series와 DataFrame은 내부적으로 NumPy 배열을 사용한다. 따라서 Pandas를 사용하면 NumPy의 빠른 연산 성능을 자연스럽게 활용하게 된다.

**NumPy와 Pandas의 관계:**

```
┌──────────────────────────────────────────────────────┐
│                    Pandas                             │
│  ┌────────────┐  ┌────────────────────────────────┐  │
│  │  Series    │  │  DataFrame                     │  │
│  │ (1차원)    │  │  (2차원 = 여러 Series 결합)    │  │
│  └─────┬──────┘  └──────────────┬─────────────────┘  │
│        │                        │                     │
│        └──────────┬─────────────┘                     │
│                   ▼                                   │
│  ┌──────────────────────────────────────────────┐    │
│  │              NumPy Array                      │    │
│  │       (고속 수치 연산 엔진, C언어 기반)         │    │
│  └──────────────────────────────────────────────┘    │
└──────────────────────────────────────────────────────┘

→ Pandas = NumPy + 인덱스/컬럼명 + 편리한 데이터 조작 기능
→ NumPy가 "엔진"이라면, Pandas는 "자동차 전체"
```

**Pandas란?**

Pandas는 Python에서 데이터 분석을 위한 핵심 라이브러리이다. NumPy 기반으로 만들어졌으며, 엑셀처럼 표 형태의 데이터를 다루면서도 수십만~수백만 행의 데이터를 빠르게 처리할 수 있다.

**왜 Pandas를 사용하는가?**
- 엑셀로는 1만 행 이상의 데이터 처리가 느리고 한계가 있다
- SQL처럼 필터링, 집계가 가능하면서도 Python의 유연성을 갖는다
- 데이터 읽기(CSV, Excel, JSON, DB) → 전처리 → 분석 → 시각화까지 한 번에 가능하다
- scikit-learn, Streamlit 등 다른 라이브러리와 자연스럽게 연동된다

Pandas는 두 가지 기본 자료구조를 제공한다.

| 자료구조 | 차원 | 설명 | 비유 |
|----------|------|------|------|
| **Series** | 1차원 | 인덱스 + 값으로 구성된 배열 | 엑셀의 한 열(column) |
| **DataFrame** | 2차원 | 행(row) × 열(column)로 구성된 테이블 | 엑셀 시트 전체 |

`pandas_basics.py` 파일 생성 (경로: `bigdata-project/2w/pandas_basics.py`):

```python
import pandas as pd

# ── Series 생성 ──
scores = pd.Series([85, 92, 78, 95], index=["철수", "영희", "민수", "수진"])
print(scores)
# 철수    85
# 영희    92
# 민수    78
# 수진    95

# Series는 다양한 통계 메서드를 제공한다
print(scores.mean())     # 87.5  (평균)
print(scores.sum())      # 350   (합계)
print(scores.max())      # 95    (최댓값)
print(scores["영희"])    # 92    (인덱스로 접근)
print(scores[scores >= 90])  # 90점 이상만 필터링

# ── DataFrame 생성 ──
data = {
    "이름": ["철수", "영희", "민수"],
    "국어": [85, 92, 78],
    "수학": [90, 88, 95]
}
df = pd.DataFrame(data)
print(df)
#   이름  국어  수학
# 0  철수   85   90
# 1  영희   92   88
# 2  민수   78   95

# DataFrame에서 한 열을 선택하면 Series가 된다
print(type(df["국어"]))   # <class 'pandas.core.series.Series'>
print(df["국어"].mean())  # 85.0
```

### 1-2. DataFrame의 구조 이해

```
         국어  수학  영어     ← 열(columns) = 변수(Feature/Variable)
    0     85   90   88     ← 행(row) = 하나의 관측값(데이터 1건)
    1     92   88   95     ← 행(row)
    2     78   95   82     ← 행(row)
    ↑
  인덱스(index)
```

- **인덱스(index)**: 각 행을 고유하게 식별하는 레이블 (기본값: 0, 1, 2, ...)
- **컬럼(columns)**: 각 열의 이름 (= 변수명, 피처명)
- DataFrame은 여러 Series가 열(column)로 결합된 구조이다
- 데이터 분석의 거의 모든 작업은 DataFrame 위에서 수행된다

**데이터 분석에서의 용어 대응:**

| 일상 용어 | Pandas 용어 | 머신러닝 용어 | 엑셀 용어 |
|-----------|-------------|---------------|-----------|
| 한 건의 데이터 | 행(row) | 샘플(sample) | 행 |
| 데이터의 속성 | 열(column) | 특성/피처(feature) | 열 |
| 예측하려는 값 | 타겟 컬럼 | 레이블(label) / 타겟(target) | - |
| 예측에 사용하는 값 | 나머지 컬럼들 | 특성(features) / X | - |

> 예를 들어 "상품 판매량을 예측"하는 프로젝트에서:
> - **행**: 각 상품 하나하나 (노트북, 마우스, ...)
> - **피처(X)**: 가격, 카테고리, 평점 등
> - **타겟(y)**: 판매량 (예측하려는 값)

### 1-3. 데이터 읽기/쓰기

실제 프로젝트에서는 직접 데이터를 만들기보다 파일에서 읽어온다.

**데이터 소스별 읽기 방법:**

```python
# ── CSV 파일 읽기 (가장 많이 사용) ──
df = pd.read_csv("data.csv")

# 한글이 포함된 CSV — 인코딩 문제 해결
df = pd.read_csv("data.csv", encoding="utf-8-sig")    # 보통 이것으로 해결
df = pd.read_csv("data.csv", encoding="cp949")         # 윈도우에서 만든 CSV
df = pd.read_csv("data.csv", encoding="euc-kr")        # 오래된 한글 인코딩

# ── Excel 파일 읽기 ──
df = pd.read_excel("data.xlsx")
df = pd.read_excel("data.xlsx", sheet_name="시트1")    # 특정 시트 지정
df = pd.read_excel("data.xlsx", sheet_name=0)           # 첫 번째 시트 (0부터)

# ── JSON 파일 읽기 ──
df = pd.read_json("data.json")

# ── 저장 ──
df.to_csv("output.csv", index=False, encoding="utf-8-sig")
df.to_excel("output.xlsx", index=False)
```

**인코딩 오류(UnicodeDecodeError) 발생 시 해결 순서:**

```
1단계: encoding="utf-8-sig" 시도
2단계: encoding="cp949" 시도
3단계: encoding="euc-kr" 시도
4단계: 위 모두 안 되면 → encoding="latin-1" (깨진 글자로 읽힌 후 확인)
```

> **encoding 팁**: CSV 저장 시 `encoding="utf-8-sig"`를 사용하면 엑셀에서 열어도 한글이 깨지지 않는다. `index=False`를 넣지 않으면 불필요한 인덱스 열이 추가되므로 항상 넣는 것이 좋다.

**`read_csv()`의 유용한 옵션들:**

```python
df = pd.read_csv(
    "data.csv",
    encoding="utf-8-sig",          # 인코딩
    usecols=["이름", "가격"],       # 필요한 컬럼만 읽기 (메모리 절약)
    nrows=1000,                     # 처음 1000행만 읽기 (데이터 미리보기 용)
    parse_dates=["날짜"],           # 날짜 컬럼을 자동으로 datetime 타입으로 변환
    dtype={"카테고리": "category"}, # 타입을 미리 지정 (메모리 절약)
    na_values=["", "N/A", "-"]      # 결측치로 인식할 값 지정
)
```

## 2. 데이터 탐색 — 분석의 첫 걸음

### 2-1. 왜 데이터 탐색이 중요한가?

데이터를 받으면 바로 분석이나 모델링을 시작하면 안 된다. 가장 먼저 해야 할 일은 **데이터의 구조와 품질을 파악**하는 것이다. 이 과정을 **EDA(Exploratory Data Analysis, 탐색적 자료분석)** 라고 한다.

**데이터 탐색을 건너뛰면 생기는 문제:**

```
❌ "왜 모델 정확도가 50%밖에 안 되지?"
   → 알고 보니 타겟 변수의 90%가 같은 값 (불균형 데이터)

❌ "왜 평균 가격이 5000만원이지?"
   → 알고 보니 '가격' 컬럼에 '원' 단위와 '만원' 단위가 섞여 있었음

❌ "왜 코드가 에러가 나지?"
   → 알고 보니 숫자 컬럼인데 문자열(object)로 저장되어 있었음
```

---

#### 심층사례: 정확도 99%의 함정 (정확도의 역설)

데이터 1,000개 중 스팸이 10개, 정상이 990개인 상황을 가정해 보자.
이 모델은 학습을 전혀 하지 않고 그냥 **"모든 메일은 정상입니다!"** 라고 대답하도록 프로그래밍 되었다.

- **결과**: 990개의 정상 메일은 다 맞혔고, 10개의 스팸 메일만 틀림
- **정확도 계산**: 990(맞힌 개수) / 1,000(전체 개수) × 100 = **99%**

수치상으로는 완벽해 보이지만, 정작 우리가 원했던 **'스팸을 걸러내는 기능'** 은 0%이다.
이를 **정확도의 역설(Accuracy Paradox)** 이라고 한다.

> 불균형 데이터에서는 '정확도' 대신 다른 지표를 반드시 함께 봐야 한다.

| 지표 | 의미 | 이 상황에서의 결과 |
|------|------|-------------------|
| **정밀도 (Precision)** | 스팸이라고 예측한 것 중 진짜 스팸인 비율 | **0** — 스팸이라고 예측한 적이 없음 |
| **재현율 (Recall)** | 실제 스팸들 중 스팸이라고 찾아낸 비율 | **0** — 10개 중 하나도 못 찾음 |
| **F1-Score** | 정밀도와 재현율의 조화 평균 | **0** — 진짜 성능을 드러냄 |

- **정밀도**는 "예측의 품질" → 스팸이라고 말했을 때 얼마나 믿을 수 있는가?
- **재현율**은 "탐지의 범위" → 실제 스팸을 얼마나 빠짐없이 잡아내는가?
- **F1-Score**는 둘의 균형 → 불균형 데이터에서 모델 평가의 핵심 지표

---

#### 심층사례: 단위 혼용과 수치 왜곡 (Data Inconsistency)

**1. 왜 이런 일이 발생하나요?**

보통 여러 소스에서 데이터를 긁어오거나(Crawling), 사람이 직접 입력할 때 발생한다.

```
A 소스: "억" 단위로 기재   → 5.5     (= 5억 5천만 원)
B 소스: "만 원" 단위로 기재 → 55,000  (= 5억 5천만 원)
```

컴퓨터는 이 숫자가 단위를 포함한 의미인지 모른다. 그저 `5.5`와 `55,000`이라는 숫자 자체로만 인식하여 계산해 버린다.

**2. 왜 `describe()`의 Max값만 봐도 알 수 있나요?**

Pandas의 `df.describe()`는 데이터의 기초 통계량(평균, 최소, 최대 등)을 한눈에 보여주는 도구다.

- **정상적인 데이터**: Max값이 평균(Mean)이나 75% 지점에서 크게 벗어나지 않는다.
- **단위 혼용 데이터**: 대부분이 '억' 단위인데 혼자 '원' 단위로 입력된 데이터가 있다면, Max값이 상상을 초월하는 숫자로 튀어 오른다.

> 예: 서울 아파트 평균이 10억인데, 어떤 데이터가 "10억" 대신 "1,000,000,000(원)"으로 적혀 있다면 Max값은 10억이 아니라 100억 단위를 훌쩍 넘게 찍힌다.

**3. "수치가 무의미해진다"는 의미 (Garbage In, Garbage Out)**

통계의 기본인 **평균(Mean)** 은 극단적인 값(Outlier)에 매우 취약하다.

예시: 5억짜리 집 9채와 5만 원으로 잘못 기재된 집 1채가 있다면?

```
실제 평균: 약 4억 5천만 원
계산된 평균: (500,000,000 × 9 + 50,000) / 10 = 450,005,000
```

이건 그나마 나은 경우다. 반대로 **5억을 50,000(만 원)** 으로 적어야 하는데 실수로 **500,000,000(만 원)** 으로 적었다면? 평균 가격이 수십 배로 뻥튀기되어, 분석 결과 전체가 오염된다.

**4. 해결 방법 (Data Cleaning)**

이런 문제는 모델을 돌리기 전 전처리(Preprocessing) 단계에서 반드시 잡아내야 한다.

| 방법 | 설명 |
|------|------|
| **시각화** | Boxplot을 그려서 유난히 멀리 떨어져 있는 점(이상치)이 있는지 확인 |
| **단위 통일** | 특정 기준(예: 만 원)을 정해놓고 전체 데이터를 나누거나 곱해서 맞춤 |
| **도메인 지식 활용** | "서울 아파트가 5만 원일 리가 없다"는 상식을 필터로 걸어 비정상 데이터를 제거/수정 |

> **Tip**: 데이터의 Max값과 Min값이 내 상식과 일치하는지 확인하는 것이 데이터 사이언티스트의 첫 번째 의무다.

---

**데이터 탐색의 체계적 접근법 (5단계):**

```
① 크기 확인     → df.shape         "데이터가 몇 행, 몇 열인가?"
② 미리보기      → df.head()        "데이터가 어떻게 생겼는가?"
③ 타입/결측 확인 → df.info()        "각 컬럼의 타입과 빈 값은?"
④ 통계 요약     → df.describe()    "수치의 분포와 범위는?"
⑤ 분포 확인     → value_counts()   "범주별 비율은?"
```

데이터 탐색을 통해 다음을 확인한다:
- 데이터가 몇 행, 몇 열인지 (규모)
- 각 컬럼의 데이터 타입이 올바른지 (숫자가 문자열로 되어 있진 않은지)
- 결측치(비어있는 값)가 얼마나 있는지 (비율은?)
- 수치형 데이터의 분포는 어떤지 (평균, 최솟값, 최댓값, 치우침)
- 이상한 값(이상치)이 있는지 (min/max가 상식적인 범위인지)
- 범주형 데이터의 분포는 어떤지 (특정 범주에 몰려 있진 않은지)

### 2-2. 데이터 탐색 메서드 정리

| 메서드 | 역할 | 확인 포인트 |
|--------|------|-------------|
| `df.shape` | (행 수, 열 수) 튜플 반환 | 데이터 규모 파악 |
| `df.head()` | 처음 5행 표시 | 데이터 형태 미리보기 |
| `df.tail()` | 마지막 5행 표시 | 데이터 끝부분 확인 |
| `df.head(10)` | 처음 10행 표시 | 숫자 지정 가능 |
| `df.info()` | 컬럼별 데이터 타입, Non-Null 수 | 타입 이상, Null 존재 여부 |
| `df.describe()` | 수치형 컬럼 기술통계 | 평균, 표준편차, 최솟값, 최댓값, 사분위수 |
| `df.dtypes` | 각 컬럼의 데이터 타입 | object, int64, float64 등 확인 |
| `df.columns` | 컬럼명 목록 (Index 객체) | 오타, 공백 포함 여부 확인 |
| `df.nunique()` | 컬럼별 고유값 수 | 범주형 변수 판별 기준 |
| `df["col"].value_counts()` | 값별 빈도 (내림차순) | 범주 분포 확인 |

**`df.info()` vs `df.describe()` — 역할이 완전히 다르다:**

| 구분 | `df.info()` | `df.describe()` |
|------|-------------|-----------------|
| **확인하는 것** | **형식적 무결성** | **내용적 타당성** |
| 핵심 질문 | "데이터 형태가 올바른가?" | "데이터 값이 합리적인가?" |
| 확인 사항 | 타입 오류, 결측치 존재, 메모리 | 평균/최솟값/최댓값의 상식성 |
| 비유 | 이력서 형식 검토 (사진 있는지, 항목 빈칸 없는지) | 이력서 내용 검토 (경력이 말이 되는지) |

> **실용 팁**: `df.describe()`의 **max 값을 반드시 확인**하라. max가 비상식적으로 크면 이상치가 숨어있다는 신호이다. 예를 들어 "나이" 컬럼의 max가 999라면, 누군가 결측치 대신 999를 입력한 것일 수 있다.

### 2-3. 핵심 개념 — 기술통계(Descriptive Statistics)

`df.describe()` 결과를 이해하려면 기술통계 개념을 알아야 한다. 기술통계는 데이터의 특성을 숫자로 요약하는 방법이다.

#### `df.describe()` 출력 항목별 의미

| 통계량 | 의미 | 확인할 것 |
|--------|------|-----------|
| **count** | 결측치를 제외한 유효 데이터 수 | 전체 행 수와 다르면 결측치 존재 |
| **mean** (평균) | 모든 값의 합 / 개수 | 데이터의 중심이 어디인가 |
| **std** (표준편차) | 평균으로부터 얼마나 퍼져 있는가 | 값이 크면 데이터가 넓게 분포 |
| **min** | 최솟값 | 비상식적으로 작은 값은 없는가 (음수, 0 등) |
| **25%** (Q1, 1사분위수) | 하위 25% 지점의 값 | 데이터의 하위권 |
| **50%** (Q2, 중앙값/median) | 정렬했을 때 가운데 값 | mean과 차이가 크면 → 분포가 치우침 |
| **75%** (Q3, 3사분위수) | 하위 75% 지점의 값 | 데이터의 상위권 |
| **max** | 최댓값 | 비상식적으로 큰 값은 없는가 (이상치?) |

#### 핵심 통계 지표 상세 설명

**① 평균(Mean) — 데이터의 무게중심**

```python
# 평균 = 모든 값의 합 ÷ 개수
data = [85, 92, 78, 95, 88]
mean = sum(data) / len(data)   # = 438 / 5 = 87.6
```

- 가장 널리 사용되는 대표값이지만, **이상치에 매우 민감**하다
- 데이터가 대칭적으로 분포할 때 적합

**② 중앙값(Median) — 정렬했을 때 가운데 값**

```python
# 데이터를 정렬한 후 가운데 위치한 값
data = [78, 85, 88, 92, 95]   # 정렬 후
median = 88                     # 가운데(3번째) 값

# 짝수 개일 때는 가운데 두 값의 평균
data = [78, 85, 88, 92]        # 4개
median = (85 + 88) / 2          # = 86.5
```

- 이상치에 **강건(robust)** 하다
- 소득, 집값 등 극단값이 있는 데이터에서 더 적합

**③ 평균 vs 중앙값 — 핵심 비교**

```
데이터: [20, 25, 30, 28, 22, 500]   ← 500은 이상치

평균(mean):   104.2   ← 이상치(500)에 의해 크게 왜곡됨
중앙값(median): 26.5  ← 이상치 영향 거의 없음
```

| 상황 | 적합한 지표 | 이유 |
|------|-----------|------|
| 정규분포(종 모양) | 평균 | 대칭이므로 평균 = 중앙값 |
| 이상치 있음 | **중앙값** | 극단값에 영향받지 않음 |
| 분포가 치우침(skewed) | **중앙값** | 더 정확한 "중심"을 나타냄 |
| 결측치 대체 | 상황에 따라 | 이상치 의심 → 중앙값, 아니면 → 평균 |

> **실무 팁**: `df.describe()`에서 **mean과 50%(중앙값)의 차이가 크면** 이상치가 있거나 분포가 한쪽으로 치우쳐 있다는 신호이다. 이 경우 해당 컬럼을 자세히 살펴봐야 한다.

**④ 표준편차(Standard Deviation, std) — 데이터의 퍼짐 정도**

```
A반 점수: [85, 87, 86, 88, 84]  → std ≈ 1.6  (점수가 고르게 분포)
B반 점수: [50, 100, 70, 95, 60] → std ≈ 20.7 (점수 편차가 큼)
```

- 표준편차가 **크면**: 데이터가 평균으로부터 멀리 퍼져 있다 → 값의 변동이 크다
- 표준편차가 **작으면**: 데이터가 평균 주위에 모여 있다 → 값이 비슷하다
- 두 그룹을 비교할 때 유용하다 (예: A반과 B반의 성적 편차 비교)

**⑤ 사분위수(Quartile)와 IQR**

데이터를 크기 순으로 정렬했을 때 4등분하는 값이다.

```
데이터를 정렬: [3, 7, 8, 12, 15, 18, 22, 25, 30, 100]

Q1 (25%) = 8     ← 하위 25% 지점
Q2 (50%) = 16.5  ← 중앙값
Q3 (75%) = 25    ← 상위 25% 지점

IQR = Q3 - Q1 = 25 - 8 = 17   ← 중간 50%가 분포하는 범위
```

**IQR (Interquartile Range, 사분위범위) — 이상치 탐지의 기준**

```
IQR = Q3 - Q1

이상치 경계:
  하한 = Q1 - 1.5 × IQR    ← 이보다 작으면 이상치
  상한 = Q3 + 1.5 × IQR    ← 이보다 크면 이상치

위 데이터에서:
  하한 = 8 - 1.5 × 17 = -17.5
  상한 = 25 + 1.5 × 17 = 50.5
  → 100은 50.5를 넘으므로 이상치!
```

이 개념은 **박스플롯(Box Plot)** 차트의 기초이며, 이상치 탐지에서 가장 많이 사용하는 방법이다.

```
박스플롯 구조:

    ●              ← 이상치 (상한 초과)
    |
  ──┬──  상한 (Q3 + 1.5×IQR)
    |
  ┌─┤    Q3 (75%)
  │ │
  │─│    Q2 (중앙값, 50%)
  │ │
  └─┤    Q1 (25%)
    |
  ──┴──  하한 (Q1 - 1.5×IQR)
```

**⑥ 왜도(Skewness) — 분포의 치우침**

**왜도(Skewness)** 는 데이터가 어디로 '치우쳐져' 있는지를 나타내는 지표이다.

```python
print(df["가격"].skew())
```

| 값 | 의미 | 형태 |
|----|------|------|
| skew ≈ 0 | 대칭 분포 | 정규분포 형태 |
| skew > 0 | 오른쪽 꼬리 (양의 왜도) | 저가 상품이 많고 고가 상품은 소수 |
| skew < 0 | 왼쪽 꼬리 (음의 왜도) | 고가 상품이 많고 저가 상품은 소수 |

> 프로젝트에서 왜도가 크면(|skew| > 2) 로그 변환(`np.log1p()`)을 적용하면 분포가 정규분포에 가까워져 모델 성능이 향상될 수 있다.

---

#### 개발자 월급으로 이해하는 왜도

**1. 왜도 ≈ 0 (대칭 분포)**

대부분의 개발자가 비슷한 연봉(예: 5,000만 원)을 받고, 아주 적게 받는 사람과 아주 많이 받는 사람의 비율이 비슷한 경우이다.

- 형태: 종 모양의 예쁜 산맥 모양
- 특징: 평균, 중앙값, 최빈값이 거의 일치 → 분석하기 가장 편안한 상태

**2. 왜도 > 0 (오른쪽으로 긴 꼬리 / 양의 왜도)**

대부분은 연봉 3,000~5,000만 원 사이인데, **연봉 100억인 슈퍼 개발자(이상치)** 가 한 명 끼어 있는 경우이다.

- 데이터의 왜곡: 이 한 명 때문에 평균 연봉이 확 올라가 버린다. 실제 대다수 개발자의 삶은 팍팍한데, 통계상 평균은 "우리 팀 평균 연봉 1억 넘네?"라고 거짓말을 하게 된다.
- 그래프: 왼쪽으로 쏠려 있고, 오른쪽으로 꼬리가 길게 늘어진다.
- 해결책: 로그 변환을 하면 100억이라는 숫자가 작아지면서, 데이터들이 옹기종기 모여 정규분포처럼 변한다.

**3. 왜도 < 0 (왼쪽으로 긴 꼬리 / 음의 왜도)**

대부분이 1억 이상의 고연봉자인데, 열정 페이를 받는 신입 몇 명이 아주 낮은 연봉을 받는 경우이다.

- 특징: 데이터가 오른쪽에 뭉쳐 있고, 왼쪽으로 꼬리가 길게 늘어진다.
- 비유: "대부분 잘 사는데, 몇몇이 평균을 깎아먹고 있네?"

**요약 표**

| 구분 | 꼬리 방향 | 특징 | 비유 |
|------|----------|------|------|
| 양의 왜도 (>0) | 오른쪽 | 평균 > 중앙값 | "평균의 함정" (극소수의 고소득자) |
| 대칭 (≈0) | 없음 | 평균 ≈ 중앙값 | "평범하고 평화로운 상태" |
| 음의 왜도 (<0) | 왼쪽 | 평균 < 중앙값 | "대부분 고수" (극소수의 저득점자) |

---

#### 왜도에 따른 변환 — 왜 정규분포로 만들어야 하는가?

데이터 분석에서 왜도에 따라 특정 변환(Log, Square root 등)을 하는 이유는 한마디로 **"찌그러진 데이터 분포를 정규분포(종 모양)처럼 펴주기 위해서"** 이다.
머신러닝 알고리즘(특히 선형 회귀나 로지스틱 회귀)은 데이터가 정규분포를 따를 때 가장 똑똑하게 학습하기 때문이다.

**1. 양의 왜도(>0)일 때: 로그 변환(`np.log1p`)을 하는 이유**

가장 자주 만나는 케이스이다. (예: 개발자 연봉, 집값)

- 상황: 연봉 3천만 원인 사람이 대다수인데, 혼자 100억을 버는 개발자가 있다. 이 100억이라는 숫자가 너무 커서 그래프가 오른쪽으로 아주 길게 늘어진다.
- 변환의 원리: 로그(log)는 큰 숫자를 아주 작게 만드는 성질이 있다.
  - `log(10) = 1`
  - `log(1,000,000) = 6`
- 결과: 100억처럼 너무 튀는 값(이상치)을 안쪽으로 확 끌어당겨 준다. 그러면 오른쪽 꼬리가 짧아지면서 대칭인 종 모양에 가까워진다.

**2. 음의 왜도(<0)일 때: 제곱(x²)이나 지수 변환을 하는 이유**

반대 상황이다. (예: 퇴직 전 연령대, 쉬운 시험 점수)

- 상황: 대부분 90~100점인데, 0점이나 10점 맞은 학생이 몇 명 있어서 왼쪽 꼬리가 긴 경우이다.
- 변환의 원리: 숫자를 제곱(x²)하면 큰 숫자는 훨씬 더 커지고, 작은 숫자는 상대적으로 덜 커진다.
  - `10 → 100` (90 증가)
  - `100 → 10,000` (9,900 증가)
- 결과: 오른쪽에 뭉쳐 있던 큰 값들을 오른쪽으로 더 멀리 밀어버린다. 그러면 왼쪽으로 치우쳤던 분포가 오른쪽으로 펴지면서 균형을 맞추게 된다.

**3. 왜 굳이 정규분포로 만들어야 하는가? (핵심)**

| 이유 | 설명 |
|------|------|
| **이상치의 영향력 감소** | 로그 변환 등을 거치면 너무 큰 값이 모델의 판단을 흐리는 것을 방지한다 |
| **모델의 가정 충족** | 많은 통계 모델과 ML 알고리즘은 "데이터가 정규분포"라는 가정으로 만들어졌다. 가정을 맞춰주면 성능(정확도)이 올라간다 |
| **데이터 간의 간격 조정** | 너무 넓게 퍼진 데이터 간격을 좁혀주어 모델이 패턴을 더 잘 찾게 돕는다 |

### 2-4. 실습 — 데이터 탐색하기

`pandas_analysis.py` 파일 생성:

```python
import pandas as pd

# ──────────────────────────────────────
# 샘플 데이터 생성 (결측치, 이상치 포함)

data = {
```

## 3. 데이터 전처리 — 분석 가능한 상태로 만들기

### 3-0. 데이터 전처리가 왜 중요한가?

> **"데이터 분석 프로젝트에서 전체 시간의 60~80%는 전처리에 소요된다."**
> — 실제 현업 데이터 사이언티스트들의 공통된 경험

**데이터의 민낯 — 현실에서 만나는 데이터는 이렇게 생겼다:**

```
실제 부동산 데이터에서 발견되는 문제들:

  가격 컬럼:  [32000, 15000, "가격협의", 28000, 45000, "", 31000]
                                ↑                    ↑
                         숫자가 아닌 텍스트!      빈 문자열!

  → 이 "가격협의" 딱 한 개 때문에 df["가격"].mean() 전체가 에러!
  → 10만 건 중에 딱 1건만 이래도 전체 분석이 멈춘다.
```

> **핵심**: 데이터 1건의 오류가 10만 건 전체의 분석을 멈추게 할 수 있다. 이것이 전처리가 중요한 이유이다.

**전처리가 중요한 3가지 이유:**

**1. 현실의 데이터는 절대로 깨끗하지 않다**

교과서에서 다루는 데이터는 정제된 상태이지만, 실제 프로젝트에서 만나는 데이터는 아래 문제를 반드시 포함하고 있다.

| 문제 유형 | 실제 예시 | 방치 시 결과 |
|-----------|----------|-------------|
| **결측치** (비어있는 값) | 설문 응답 누락, 센서 오작동 | 분석 불가, 모델 에러 |
| **타입 오류** | "1,200,000"이 문자열로 저장 | 평균/합계 계산 불가 |
| **중복 데이터** | 같은 데이터가 여러 번 수집 | 분석 결과 왜곡 (빈도, 비율 오류) |
| **이상치** | 나이: 250세, 가격: -100원 | 평균 왜곡, 모델 성능 저하 |
| **불일치** | "서울", "서울시", "Seoul" 혼재 | 그룹핑 시 같은 값이 다르게 집계 |
| **불필요한 데이터** | 분석과 무관한 컬럼, 빈 컬럼 | 메모리 낭비, 분석 혼란 |

**2. 전처리 품질이 모델 성능을 결정한다**

```
같은 머신러닝 알고리즘을 사용해도:
  전처리를 대충 한 경우   →  정확도 60~70%  (쓸모없는 수준)
  전처리를 꼼꼼히 한 경우 →  정확도 85~95%  (실용적인 수준)
```

> 좋은 모델을 찾는 것보다 **좋은 데이터를 만드는 것**이 더 중요하다. 

**3. "Garbage In, Garbage Out"**

이 원칙은 데이터 분석의 가장 기본적인 진리이다. 아무리 좋은 알고리즘을 사용해도 **입력 데이터가 잘못되면 결과도 잘못된다.** 반대로, 데이터가 잘 정제되어 있으면 단순한 알고리즘으로도 좋은 결과를 얻을 수 있다.

**전처리 워크플로우:**

```
① 결측치 확인     "빈 값이 얼마나 있는가?"
    ↓
② 결측치 처리     "삭제? 대체? 어떤 값으로?"
    ↓
③ 데이터 타입 변환  "숫자/날짜/범주가 올바른 타입인가?"
    ↓
④ 중복 제거       "같은 데이터가 반복되어 있는가?"
    ↓
⑤ 이상치 처리     "비정상적인 값이 있는가?"
    ↓
⑥ 문자열 정제     "공백, 대소문자, 오타 문제는 없는가?"
```

### 3-1. 결측치(Missing Value) 처리

#### 결측치란?

데이터에서 값이 비어 있는 것을 결측치(Missing Value)라 한다. Pandas에서는 `NaN` (Not a Number) 또는 `None`으로 표시된다.

**결측치가 발생하는 원인:**
- 설문 조사에서 응답을 건너뜀
- 센서/장비 오작동으로 측정 실패
- 데이터 수집 시 일부 필드가 누락됨
- 여러 데이터소스를 합칠 때 컬럼이 맞지 않음
- API에서 특정 필드가 없는 경우 (선택 항목)

#### 결측치 확인 방법

```python
# 전체 결측치 수 (컬럼별)
print(df.isnull().sum())

# 결측치 비율 (%)
print((df.isnull().sum() / len(df) * 100).round(1))

# 결측치가 있는 행만 보기
print(df[df.isnull().any(axis=1)])
```

#### 결측치 처리 방법

| 방법 | 코드 | 언제 사용하나 |
|------|------|---------------|
| **행 삭제** | `df.dropna()` | 결측 비율이 매우 낮을 때 (< 5%) |
| **특정 컬럼 기준 삭제** | `df.dropna(subset=["col"])` | 핵심 컬럼의 결측만 삭제 |
| **고정값으로 채우기** | `df["col"].fillna(0)` | 0, "없음", "미입력" 등 도메인 지식 기반 |
| **평균으로 채우기** | `df["col"].fillna(df["col"].mean())` | 수치형, 정규분포일 때 |
| **중앙값으로 채우기** | `df["col"].fillna(df["col"].median())` | 수치형, 이상치가 있을 때 |
| **최빈값으로 채우기** | `df["col"].fillna(df["col"].mode()[0])` | 범주형 데이터 |
| **앞의 값으로 채우기** | `df["col"].ffill()` | 시계열 데이터 |
| **뒤의 값으로 채우기** | `df["col"].bfill()` | 시계열 데이터 |

#### 결측치 처리 전략 선택 기준

```
결측 비율에 따른 판단:

  5% 미만     → 해당 행 삭제해도 무방
  5% ~ 30%   → 적절한 값으로 대체 (imputation) 권장
  30% 이상    → 해당 컬럼(변수) 자체를 삭제 고려
```

#### 어떤 값으로 채울 것인가?

```
수치형 변수:
  이상치 없음 → 평균(mean)
  이상치 있음 → 중앙값(median) ← 더 안전한 선택

범주형 변수:
  → 최빈값(mode) 또는 "Unknown" 같은 별도 범주

시계열 데이터:
  → 앞의 값(ffill) 또는 보간법(interpolate)
```

#### 도메인 지식이 fillna의 핵심이다

결측치를 무조건 평균이나 중앙값으로 채우면 안 된다. **"이 데이터가 왜 비어있는가?"** 를 먼저 생각해야 한다. 이것이 CRISP-DM의 ① 비즈니스 이해 단계와 연결된다.

```
실제 사례: 쇼핑몰 리뷰 데이터

  상품명      평점    리뷰내용
  노트북A     5.0    "최고입니다!"
  마우스B     NaN    NaN
  키보드C     4.0    "괜찮아요"

  Q: 마우스B의 평점 NaN을 평균(4.5)으로 채워도 될까?
  A: ❌ 리뷰를 아예 안 쓴 사람 → 만족도가 낮을 가능성 높음
     → 평균보다 낮은 값(예: 3.0)이나 별도 범주("미평가")가 더 적절

  Q: 온라인 쇼핑몰에서 "배송 소요일" 컬럼이 NaN이면?
  A: 아직 배송이 안 된 건 → 0이나 평균이 아니라, 해당 행 자체를 분석 대상에서 제외

→ "왜 비어있는가?"에 따라 채우는 값이 완전히 달라진다.
→ 이것이 "도메인 지식"이며, AI가 대신할 수 없는 분석가의 역할이다.
```

### 3-2. 데이터 타입 변환

데이터 타입이 잘못되어 있으면 연산이나 시각화에서 오류가 발생한다. 특히 API나 웹 크롤링으로 수집한 데이터는 모든 것이 문자열(object)로 저장되는 경우가 많다.

**Pandas의 주요 데이터 타입:**

| Pandas 타입 | 설명 | 예시 |
|-------------|------|------|
| `int64` | 정수 | 1, 100, -5 |
| `float64` | 실수 (소수점) | 3.14, 99.9 |
| `object` | 문자열 (범주, 텍스트) | "서울", "노트북" |
| `bool` | 참/거짓 | True, False |
| `datetime64` | 날짜/시간 | 2026-03-08 |
| `category` | 범주형 (메모리 최적화) | "남", "여" |

**흔한 문제 상황과 해결법:**

```python
# ── 문제 1: 숫자인데 문자열로 저장됨 ──
# 원인: CSV에 "1,200,000"처럼 쉼표가 포함되어 있으면 object로 읽힘
#
# 변환 과정:
#   "1,200,000" (문자열, 계산 불가)
#     → str.replace(",", "") → "1200000" (쉼표 제거)
#     → pd.to_numeric()     → 1200000  (숫자, 계산 가능!)

df["가격"] = df["가격"].str.replace(",", "")    # 1단계: 쉼표 제거
df["가격"] = df["가격"].astype(int)              # 2단계: 정수로 변환

# 또는 한 번에 (권장 — 변환 불가 값도 안전하게 처리)
df["가격"] = pd.to_numeric(df["가격"].str.replace(",", ""), errors="coerce")
# errors="coerce" → 변환 불가능한 값("미정", "N/A" 등)은 에러 대신 NaN으로 처리

# ── 문제 2: 숫자 중 변환 불가한 값이 섞여 있음 ──
# 예: ["100", "200", "미입력", "300"]
df["값"] = pd.to_numeric(df["값"], errors="coerce")
# → "미입력"은 NaN으로 변환됨 (에러 없이 진행)

# ── 기본 타입 변환 ──
df["컬럼"] = df["컬럼"].astype(int)           # 정수형
df["컬럼"] = df["컬럼"].astype(float)         # 실수형
df["컬럼"] = df["컬럼"].astype(str)           # 문자열

# ── 날짜형 변환 + 특성 추출 (시계열 분석의 기초) ──
# "2024-03-15" (문자열) → datetime 타입으로 변환
df["날짜"] = pd.to_datetime(df["날짜"])

# 날짜에서 다양한 정보 추출 가능 (= 특성 엔지니어링)
df["연도"] = df["날짜"].dt.year               # 2024-03-15 → 2024
df["월"] = df["날짜"].dt.month                # 2024-03-15 → 3
df["일"] = df["날짜"].dt.day                  # 2024-03-15 → 15
df["요일"] = df["날짜"].dt.day_name()         # → "Friday"
df["주차"] = df["날짜"].dt.isocalendar().week  # → 11 (11번째 주)
df["분기"] = df["날짜"].dt.quarter            # → 1 (1분기)

# 주말 여부 판별 (특성 엔지니어링 활용 예시)
# dt.dayofweek: 월=0, 화=1, 수=2, 목=3, 금=4, 토=5, 일=6
df["주말"] = df["날짜"].dt.dayofweek.isin([5, 6])
# → 토(5) 또는 일(6)이면 True, 아니면 False
# → "주말에 매출이 높은가?" 같은 분석에 활용

# ── 범주형 변환 (대용량 데이터에서 특히 효과적) ──
df["카테고리"] = df["카테고리"].astype("category")
# → 고유값 목록(사전)을 한 번만 저장, 각 행에는 정수 코드만 기록
# → 메모리 절약 + groupby 성능 향상
# → 예: 10만행, 고유값 5개 → 메모리 90% 이상 절약
```

> **타입 변환 시 주의**: `astype(int)`는 NaN이 있으면 에러가 난다. NaN을 먼저 처리한 후에 타입 변환을 해야 한다. 또는 `pd.to_numeric(errors="coerce")`를 사용하면 안전하다. 실무에서는 평균 자체가 NaN인 경우(전부 결측인 컬럼)도 있기 때문에, `astype(int)` 전에 반드시 결측/이상치를 모두 처리하고 변환하는 습관이 필요하다.

### 3-3. 중복 데이터 처리

같은 데이터가 여러 번 들어있으면 분석 결과가 왜곡된다. 특히 **여러 소스에서 데이터를 합칠 때** 중복이 자주 발생한다.

**중복이 발생하는 흔한 원인:**
- 데이터 수집 스크립트를 여러 번 실행
- 여러 CSV 파일을 합칠 때 중첩 구간 존재
- API 호출 시 페이지 경계에서 같은 데이터가 반복

```python
# ── 중복 확인 ──
print(f"중복 행: {df.duplicated().sum()}개")
print(f"중복 비율: {df.duplicated().sum() / len(df) * 100:.1f}%")

# 중복 행 보기 (원본과 중복 모두 표시)
print(df[df.duplicated(keep=False)])

# 어떤 컬럼 기준으로 중복인지 확인
print(df[df.duplicated(subset=["상품명"], keep=False)])

# ── 중복 제거 ──
# 완전히 동일한 행 제거 (첫 번째만 남김)
df = df.drop_duplicates()

# 특정 컬럼 기준 중복 제거 (나머지 컬럼이 달라도 제거)
df = df.drop_duplicates(subset=["상품명", "카테고리"])

# 마지막 행을 남기고 중복 제거 (최신 데이터를 남기고 싶을 때)
df = df.drop_duplicates(subset=["상품명"], keep="last")

# 중복 제거 후 인덱스 정리
df = df.drop_duplicates().reset_index(drop=True)
```

> **주의**: `drop_duplicates()`는 기본적으로 **모든 컬럼이 동일한 행**만 중복으로 판단한다. 특정 컬럼만 기준으로 하려면 반드시 `subset` 파라미터를 지정해야 한다.

### 3-4. 이상치(Outlier) 탐지 및 처리

#### 이상치란?

다른 데이터와 비교했을 때 비정상적으로 크거나 작은 값이다.

```
예: 학생 시험 점수
    [82, 75, 90, 88, 95, 3, 79, 85]
                          ↑
                     3점 = 이상치 (입력 오류? 실제 점수?)
```

**이상치의 유형:**

| 유형 | 설명 | 예시 | 처리 방법 |
|------|------|------|-----------|
| **입력 오류** | 잘못 입력된 값 | 나이: 250세, 몸무게: 0kg | 수정 또는 삭제 |
| **측정 오류** | 장비/센서 오작동 | 갑자기 온도 -999도 | 삭제 또는 대체 |
| **자연 이상치** | 실제로 존재하는 극단값 | 연봉 100억, 초고가 아파트 | 주의: 함부로 삭제하면 안 됨 |
| **표본 이상치** | 모집단과 다른 그룹의 데이터 | 성인 데이터에 유아 데이터 혼입 | 분리 처리 |

이상치가 있으면 **평균이 왜곡**되고, **머신러닝 모델의 성능이 떨어질** 수 있다. 하지만 자연 이상치는 중요한 정보를 담고 있을 수 있으므로 도메인 지식을 바탕으로 신중하게 판단해야 한다.

#### IQR 방식 이상치 탐지 (가장 많이 사용)

```python
# IQR 계산
Q1 = df["가격"].quantile(0.25)     # 1사분위수 (25%)
Q3 = df["가격"].quantile(0.75)     # 3사분위수 (75%)
IQR = Q3 - Q1                      # 사분위범위

# 이상치 경계 설정
lower_bound = Q1 - 1.5 * IQR       # 하한
upper_bound = Q3 + 1.5 * IQR       # 상한

print(f"정상 범위: {lower_bound:,.0f} ~ {upper_bound:,.0f}")

# 이상치 확인
outliers = df[(df["가격"] < lower_bound) | (df["가격"] > upper_bound)]
print(f"이상치 수: {len(outliers)}개")
print(outliers)
```

#### 이상치 처리 방법

```python
# 방법 1: 이상치 행 제거
df_clean = df[(df["가격"] >= lower_bound) & (df["가격"] <= upper_bound)]

# 방법 2: 이상치를 경계값으로 대체 (Capping/Clipping)
df["가격"] = df["가격"].clip(lower=lower_bound, upper=upper_bound)

# 방법 3: 이상치를 NaN으로 변환 후 대체
df.loc[df["가격"] > upper_bound, "가격"] = None
df["가격"] = df["가격"].fillna(df["가격"].median())
```

> **주의**: 이상치가 항상 "잘못된 데이터"는 아니다. 도메인 지식을 바탕으로 판단해야 한다. 예를 들어 고가의 노트북 가격은 이상치로 탐지될 수 있지만, 실제로 유효한 데이터이다.

**이상치 처리 판단 기준:**

```
1. 도메인 상식으로 판단
   → 나이가 0 미만이거나 150 초과? → 입력 오류 → 제거/수정
   → 가격이 음수? → 입력 오류 → 제거/수정

2. 통계적으로 판단 (IQR 또는 Z-score)
   → IQR 방식으로 탐지 후 도메인 지식과 비교
   → 무조건 제거하지 말고, "왜 이상치인지" 파악 먼저

3. 시각화로 확인
   → 박스플롯, 히스토그램으로 분포 확인 (3주차에서 상세히)
```

> **실무 팁**: IQR 기준만으로 자르는 것보다, "나이는 0~100세", "가격은 0 이상" 같은 **비즈니스 규칙**을 함께 사용하면 더 안전하게 이상치를 정의할 수 있다. 통계적 기준(IQR)과 도메인 기준을 병행하는 것이 가장 현실적인 접근법이다.

### 3-5. 문자열 데이터 정제

**왜 문자열 정제가 중요한가?**

데이터 분석에서 숫자 결측치나 이상치에 비해 문자열 문제는 간과되기 쉽지만, 실제로는 **분석 결과를 완전히 망가뜨릴 수 있는 치명적인 문제**이다.

```
문자열 정제를 하지 않으면 발생하는 문제:

1. 그룹핑 오류
   "서울", " 서울", "서울시", "서울특별시" → 4개의 별도 그룹으로 집계
   → 실제로는 같은 "서울"인데 매출이 4등분되어 잘못된 분석 결과 도출

2. 병합(merge) 실패
   A 테이블: "Samsung"  /  B 테이블: "samsung"
   → pd.merge() 시 매칭되지 않아 데이터 누락 (NaN 발생)

3. 머신러닝 성능 저하
   원-핫 인코딩 시 "서울", "서울시"가 별도 피처로 생성
   → 불필요한 차원 증가, 모델 학습 방해

4. 시각화 왜곡
   차트에서 같은 항목이 여러 개로 분리되어 표시
   → 잘못된 인사이트 도출
```

> **핵심**: 전처리의 80%는 "데이터를 일관된 형태로 통일하는 것"이다. 문자열 정제는 그 핵심 작업 중 하나이며, 특히 공공데이터나 웹 크롤링 데이터처럼 여러 출처에서 수집된 데이터일수록 반드시 수행해야 한다.

공공데이터나 웹 크롤링 데이터에서 자주 발생하는 문제이다.

```python
# ── 공백 제거 ──
df["도시"] = df["도시"].str.strip()           # 양쪽 공백 제거
df["도시"] = df["도시"].str.replace(" ", "")  # 모든 공백 제거

# ── 대소문자 통일 ──
df["이름"] = df["이름"].str.lower()           # 소문자로
df["이름"] = df["이름"].str.upper()           # 대문자로
df["이름"] = df["이름"].str.title()           # 첫 글자만 대문자

# ── 불일치 값 통일 ──
# "서울", "서울시", "서울특별시" → "서울"로 통일
df["도시"] = df["도시"].replace({
    "서울시": "서울",
    "서울특별시": "서울",
    "부산시": "부산",
    "부산광역시": "부산"
})

# ── 특수문자 제거 ──
df["텍스트"] = df["텍스트"].str.replace(r"[^\w\s]", "", regex=True)

# ── 특정 패턴 추출 ──
# 전화번호에서 지역번호 추출
df["지역번호"] = df["전화번호"].str.extract(r"(\d{2,3})-")
```

**`strip()` vs `replace()` — 언제 무엇을 쓰는가:**

```
strip()  → 양쪽 "끝"의 공백만 제거   "  서울  " → "서울"
replace() → 문자열 "내부"의 특정 값 변경  "서 울" → "서울", "서울시" → "서울"

┌───────────────────────────────────────┐
│ "  서울시  "                           │
│   ↓ strip()                           │
│ "서울시"        ← 양끝 공백만 제거     │
│   ↓ replace("시", "")                 │
│ "서울"          ← 내부 문자 치환       │
└───────────────────────────────────────┘

→ 실무에서는 strip() 먼저, 그다음 replace()를 적용하는 것이 정석이다.
```

> 문자열 정제를 하지 않으면 `groupby()`에서 "서울"과 "서울시"가 다른 그룹으로 집계된다. 1만 건 이상의 데이터에서는 이런 불일치를 육안으로 찾기 어려우므로 `value_counts()`로 반드시 확인해야 한다.

## 4. 데이터 가공 — 분석에 필요한 형태로 변환하기

전처리가 끝난 깨끗한 데이터를 분석 목적에 맞게 변환하는 단계이다.

### 4-1. 인덱싱과 슬라이싱 — 원하는 데이터 선택하기

```python
# ── 열(column) 선택 ──
df["가격"]                          # 한 열 선택 → Series
df[["상품명", "가격"]]              # 여러 열 선택 → DataFrame

# ── 행(row) 선택 ──
df.iloc[0]                          # 0번째 행 (위치 기반)
df.iloc[0:3]                        # 0~2번째 행
df.iloc[0, 1]                       # 0번째 행, 1번째 열의 값

df.loc[0]                           # 인덱스가 0인 행 (레이블 기반)
df.loc[0:2, "상품명":"가격"]        # 범위 선택 (끝 포함)
```

**iloc vs loc 차이점:**

| 구분 | `iloc` | `loc` |
|------|--------|-------|
| 기준 | 정수 위치 (0, 1, 2, ...) | 인덱스 레이블 |
| 범위 끝 | **미포함** (파이썬 슬라이싱과 동일) | **포함** |
| 사용 시기 | 위치로 접근할 때 | 인덱스/컬럼명으로 접근할 때 |

### 4-2. 필터링 — 조건에 맞는 데이터 추출

프로젝트에서 가장 자주 사용하는 작업 중 하나이다.

```python
# ── 단일 조건 ──
expensive = df[df["가격"] >= 100000]
print(expensive)

# ── 복합 조건 ──
# AND 조건: & 사용 (각 조건을 괄호로 감싸야 함!)
good_expensive = df[(df["가격"] >= 100000) & (df["평점"] >= 4.0)]

# OR 조건: | 사용
cheap_or_good = df[(df["가격"] < 50000) | (df["평점"] >= 4.5)]

# NOT 조건: ~ 사용
not_etc = df[~(df["카테고리"] == "주변기기")]

# ── 특정 값 포함 여부 ──
targets = ["컴퓨터", "주변기기"]
filtered = df[df["카테고리"].isin(targets)]

# ── 문자열 조건 ──
df[df["상품명"].str.contains("키")]          # "키"가 포함된 상품
df[df["상품명"].str.startswith("노")]        # "노"로 시작
df[df["상품명"].str.len() >= 3]              # 이름이 3자 이상

# ── 범위 조건 ──
df[df["가격"].between(50000, 500000)]        # 5만~50만 사이
```

> **자주 하는 실수**: 복합 조건에서 `and` / `or` 대신 `&` / `|`를 사용해야 한다. 그리고 각 조건을 **반드시 괄호**로 감싸야 한다.

### 4-4. 그룹핑 및 집계 — 범주별 요약 통계

**프로젝트에서 가장 많이 사용하는 패턴** 중 하나이다. 범주(그룹)별로 데이터를 요약할 때 사용한다. SQL의 `GROUP BY`와 동일한 개념이다.

**groupby의 동작 원리 (Split → Apply → Combine):**

```
원본 데이터:               Split (나누기)           Apply (적용)     Combine (합치기)
┌──────────┬──────┐
│ 카테고리  │ 가격  │      컴퓨터:                  컴퓨터:           ┌──────────┬──────────┐
│ 컴퓨터   │ 120만 │   → [120만, 45만, 80만]   → mean = 81.7만     │ 카테고리  │ 평균가격  │
│ 주변기기  │  3.5만│      주변기기:                주변기기:          │ 컴퓨터   │ 81.7만   │
│ 주변기기  │  8.9만│   → [3.5만, 8.9만, 5.5만] → mean = 6.0만      │ 주변기기  │ 6.0만    │
│ 컴퓨터   │  45만 │                                                └──────────┴──────────┘
│ 주변기기  │  5.5만│
│ 컴퓨터   │  80만 │
└──────────┴──────┘
```

```python
# ── 기본 그룹핑 ──
# 카테고리별 평균 가격
df.groupby("카테고리")["가격"].mean()

# 카테고리별 건수
df.groupby("카테고리")["가격"].count()

# ── agg()로 여러 집계를 한 번에 ──
category_stats = df.groupby("카테고리").agg(
    상품수=("상품명", "count"),
    평균가격=("가격", "mean"),
    최고가=("가격", "max"),
    총판매량=("판매량", "sum"),
    평균평점=("평점", "mean")
).round(1)
print(category_stats)

# ── 다중 그룹핑 ──
# 카테고리 × 가격대별 집계
df.groupby(["카테고리", "가격대"]).agg(
    상품수=("상품명", "count"),
    평균가격=("가격", "mean")
)
```

**`agg()`에서 사용 가능한 집계 함수:**

| 함수 | 설명 | 예시 결과 |
|------|------|-----------|
| `count` | 개수 (결측치 제외) | 5 |
| `sum` | 합계 | 1,500,000 |
| `mean` | 평균 | 300,000 |
| `median` | 중앙값 | 250,000 |
| `min` / `max` | 최솟값 / 최댓값 | 35,000 / 1,200,000 |
| `std` | 표준편차 | 150,000 |
| `first` / `last` | 첫 번째 / 마지막 값 | - |
| `nunique` | 고유값 수 | 3 |

```python
# ── 피벗 테이블 (groupby의 확장) ──
# 엑셀의 피벗 테이블과 같은 기능
# 행: 카테고리, 열: 가격대, 값: 판매량의 합계
pivot = df.pivot_table(
    values="판매량",       # 집계할 값
    index="카테고리",      # 행 기준
    columns="가격대",      # 열 기준
    aggfunc="sum",         # 집계 함수
    fill_value=0           # 빈 칸은 0으로
)
print(pivot)
# 가격대   저가  중저가  중고가  고가
# 카테고리
# 컴퓨터      0      0    200   230
# 주변기기   970    300      0     0
```

### 4-4a. 데이터 결합 — 여러 데이터를 하나로 합치기

프로젝트에서 여러 데이터소스를 결합해야 하는 경우가 많다.

**① concat — 위아래로 합치기 (같은 구조의 데이터)**

```python
# 월별로 나뉜 CSV를 하나로 합치기
df_jan = pd.read_csv("data_2026_01.csv")
df_feb = pd.read_csv("data_2026_02.csv")
df_mar = pd.read_csv("data_2026_03.csv")

df_all = pd.concat([df_jan, df_feb, df_mar], ignore_index=True)
# ignore_index=True: 인덱스를 0부터 다시 매김
print(f"합친 후: {df_all.shape}")
```

**② merge — 공통 키로 합치기 (다른 구조의 데이터)**

```python
# 상품 정보 + 판매 정보를 상품코드로 합치기
df_products = pd.DataFrame({
    "상품코드": ["A01", "A02", "A03"],
    "상품명": ["노트북", "마우스", "키보드"],
    "카테고리": ["컴퓨터", "주변기기", "주변기기"]
})

df_sales = pd.DataFrame({
    "상품코드": ["A01", "A01", "A02", "A03"],
    "날짜": ["2026-01-01", "2026-01-02", "2026-01-01", "2026-01-03"],
    "판매량": [10, 5, 30, 15]
})

# 공통 컬럼(상품코드)으로 합치기
df_merged = pd.merge(df_products, df_sales, on="상품코드")
print(df_merged)
```

**merge의 종류 (SQL JOIN과 동일):**

| how | 설명 | 결과 |
|-----|------|------|
| `inner` (기본값) | 양쪽 모두에 있는 키만 | 교집합 |
| `left` | 왼쪽 DataFrame 기준 | 왼쪽은 모두 유지 |
| `right` | 오른쪽 DataFrame 기준 | 오른쪽은 모두 유지 |
| `outer` | 양쪽 모두 포함 | 합집합 |

```python
# 왼쪽 기준 합치기 (왼쪽 데이터는 모두 유지, 오른쪽에 없으면 NaN)
df_merged = pd.merge(df_products, df_sales, on="상품코드", how="left")
```

### 4-5. 새로운 컬럼 생성 — 특성 엔지니어링(Feature Engineering)

기존 데이터를 조합하거나 변환하여 새로운 변수(컬럼)를 만드는 것이다. 이를 **특성 엔지니어링(Feature Engineering)** 이라 하며, 머신러닝 프로젝트의 성능을 좌우하는 핵심 작업이다.

```python
# ── 기존 컬럼 조합 ──
df["매출액"] = df["가격"] * df["판매량"]

# ── 구간(bin) 분류: pd.cut() ──
df["가격대"] = pd.cut(
    df["가격"],
    bins=[0, 50000, 200000, 500000, float("inf")],
    labels=["저가", "중저가", "중고가", "고가"]
)

# ── 동일 개수 분류: pd.qcut() ──
df["판매량등급"] = pd.qcut(
    df["판매량"],
    q=3,
    labels=["하위", "중위", "상위"]
)

# ── 조건에 따른 값 할당: apply() + lambda ──
df["평점등급"] = df["평점"].apply(
    lambda x: "우수" if x >= 4.5 else ("양호" if x >= 4.0 else "보통")
)

# ── 여러 컬럼을 참조하는 변환: apply(axis=1) ──
df["종합점수"] = df.apply(
    lambda row: row["평점"] * 20 + row["판매량"] / 100,
    axis=1
).round(1)
```

**특성 엔지니어링의 창의성 — 같은 데이터에서 새로운 인사이트 뽑기:**

```python
# ── 날짜 데이터에서 숨겨진 패턴 추출 ──
df["주문날짜"] = pd.to_datetime(df["주문날짜"])
df["주말여부"] = df["주문날짜"].dt.dayofweek.isin([5, 6]).astype(int)
# → 주말 주문이 평일보다 판매량이 높은가? → 프로모션 전략에 활용

# ── 생년월일에서 세대 분류 ──
df["출생연도"] = pd.to_datetime(df["생년월일"]).dt.year
df["세대"] = pd.cut(
    df["출생연도"],
    bins=[0, 1964, 1980, 1996, 2012, 2100],
    labels=["베이비부머", "X세대", "밀레니얼", "Z세대", "알파세대"]
)
# → "MZ세대의 구매 패턴은?" 이라는 분석 질문에 답할 수 있게 됨

# ── 텍스트 길이로 특성 생성 ──
df["리뷰길이"] = df["리뷰내용"].str.len()
df["상세리뷰여부"] = (df["리뷰길이"] > 100).astype(int)
# → 상세 리뷰를 쓴 사용자의 평점이 더 높은가?
```

> **핵심**: 특성 엔지니어링은 "코딩 기술"보다 **"이 데이터에서 무엇을 더 뽑아낼 수 있을까?"** 라는 창의적 사고가 더 중요하다. 좋은 특성 하나가 복잡한 모델보다 성능을 더 크게 올릴 수 있다.

**`pd.cut()` vs `pd.qcut()` 차이:**

| 구분 | `pd.cut()` | `pd.qcut()` |
|------|-----------|------------|
| 기준 | 구간 경계를 직접 지정 | 동일한 개수로 나눔 |
| 용도 | 도메인 기반 분류 | 균등 분포 분류 |
| 예시 | 0~50만=저가, 50~100만=고가 | 상위/중위/하위 3등분 |

### 5-1. 데이터 분석 프로세스 — CRISP-DM

데이터 분석/머신러닝 프로젝트의 표준 방법론이다. 이번 학기 프로젝트는 이 과정을 처음부터 끝까지 수행한다.

```
┌──────────────────────────────────────────────────────┐
│                                                      │
│   ① 비즈니스 이해     "무엇을 분석할 것인가?"         │
│          ↓                                           │
│   ② 데이터 이해       "데이터가 어떻게 생겼는가?"     │
│          ↓                                           │
│   ③ 데이터 준비       "전처리 + 특성 엔지니어링"      │
│          ↓                                           │
```

### 5-2. 변수(Feature)의 종류

데이터의 각 컬럼(변수)은 타입에 따라 다르게 분석해야 한다.

| 대분류 | 소분류 | 설명 | 예시 | 적합한 분석/시각화 |
|--------|--------|------|------|-------------------|
| **수치형** | 연속형(Continuous) | 소수점이 있는 연속적 값 | 가격, 온도, 키, 몸무게 | 평균, 표준편차, 히스토그램, 산점도 |
| | 이산형(Discrete) | 정수로 표현되는 셀 수 있는 값 | 판매량, 방 개수, 횟수 | 빈도표, 막대그래프 |
| **범주형** | 명목형(Nominal) | 순서 없이 분류만 하는 값 | 성별, 카테고리, 도시 | 빈도, 최빈값, 파이차트, 막대그래프 |
| | 순서형(Ordinal) | 순서(크기 관계)가 있는 분류 | 등급(상/중/하), 학점, 만족도 | 빈도, 중앙값, 막대그래프 |

**왜 구분이 중요한가?**
- 수치형 → 평균, 상관관계 등 수학적 연산 가능
- 범주형 → 빈도, 비율 등으로 분석 / 머신러닝 시 인코딩 필요
- **잘못된 분석 예**: "우편번호의 평균" → 숫자지만 범주형이므로 평균에 의미 없음
- **잘못된 분석 예**: "성별(1=남, 2=여)의 평균 = 1.5" → 숫자이지만 범주형이므로 의미 없음

**변수 타입을 판별하는 방법:**

```python
# 고유값 수로 판별하는 실용적 기준
for col in df.columns:
    n_unique = df[col].nunique()
    dtype = df[col].dtype
    print(f"{col}: 고유값 {n_unique}개, 타입 {dtype}")

    # 고유값이 10개 이하인 수치형 → 범주형일 가능성 높음
    if dtype in ['int64', 'float64'] and n_unique <= 10:
        print(f"  → '{col}'은 범주형으로 처리하는 것이 적합할 수 있음")
```

**범주형 인코딩 — 머신러닝 모델에 넣기 위한 변환:**

머신러닝 모델은 **숫자만 입력**으로 받을 수 있다. 따라서 "컴퓨터", "주변기기" 같은 문자열을 숫자로 변환해야 한다.

```python
# ── 방법 1: 원-핫 인코딩 (One-Hot Encoding) ──
# 범주를 0/1 컬럼으로 변환 (가장 많이 사용)
df_encoded = pd.get_dummies(df, columns=["카테고리"])
print(df_encoded.head())
# → 카테고리_컴퓨터: [1, 0, 0, 1, ...]
# → 카테고리_주변기기: [0, 1, 1, 0, ...]

# ── 방법 2: 레이블 인코딩 (Label Encoding) ──
# 순서가 있는 범주에 적합 (상/중/하 → 2/1/0)
from sklearn.preprocessing import LabelEncoder
le = LabelEncoder()
df["카테고리_코드"] = le.fit_transform(df["카테고리"])
# 컴퓨터 → 0, 주변기기 → 1
```

| 인코딩 방법 | 사용 시기 | 장점 | 단점 |
|------------|----------|------|------|
| **원-핫 인코딩** | 명목형 (순서 없음) | 범주 간 순서 관계 없음 | 고유값이 많으면 컬럼 폭발 |
| **레이블 인코딩** | 순서형 (순서 있음) | 컬럼 1개 유지 | 모델이 크기 관계로 오해할 수 있음 |

### 5-3. 상관관계(Correlation) 분석

두 수치형 변수 사이의 **선형 관계**를 수치로 나타낸 것이다. 프로젝트에서 "어떤 변수가 타겟에 영향을 미치는가?"를 파악할 때 가장 먼저 사용하는 방법이다.

**피어슨 상관계수(r) — 시각적 이해:**

```
r = +1.0 (완벽한 양의 상관)     r = 0 (상관 없음)      r = -1.0 (완벽한 음의 상관)

  y │    ●●●                   y │  ● ●  ●            y │ ●●●
    │  ●●●                       │●  ●  ●  ●            │   ●●●
    │●●●                         │ ●   ●●                │     ●●●
    └──────── x                  └──────── x              └──────── x

  키가 크면 몸무게도 무겁다     키와 혈액형은 관계없다     가격이 높으면 판매량 감소
```

| |r| 범위 | 해석 | 프로젝트에서의 의미 |
|----------|------|---------------------|
| 0.7 ~ 1.0 | **강한** 상관관계 | 중요한 피처 후보 |
| 0.4 ~ 0.7 | **중간** 상관관계 | 유용한 피처 |
| 0.2 ~ 0.4 | **약한** 상관관계 | 추가 검토 필요 |
| 0.0 ~ 0.2 | 거의 상관 없음 | 피처 제외 고려 |

```python
# ── 상관관계 행렬 ──
corr_matrix = df[["가격", "판매량", "평점"]].corr().round(2)
# 또는 df.corr(numeric_only=True) 로 숫자 컬럼만 자동 선택
print(corr_matrix)
#          가격    판매량   평점
# 가격     1.00   -0.65   0.72
# 판매량  -0.65    1.00   0.15
# 평점     0.72    0.15   1.00

# 해석:
# 가격↔평점: r=0.72 (강한 양의 상관) → 가격이 높은 상품이 평점도 높은 경향
# 가격↔판매량: r=-0.65 (중간 음의 상관) → 가격이 높으면 판매량이 낮은 경향
# 판매량↔평점: r=0.15 (거의 무관) → 판매량과 평점은 관계 약함

# ── 특정 타겟과의 상관관계만 보기 ──
# 판매량을 예측하고 싶다면:
print(df.corr(numeric_only=True)["판매량"].sort_values(ascending=False))

# ── 시각화 (3주차 이후 자세히) ──
import seaborn as sns
import matplotlib.pyplot as plt

plt.rcParams["font.family"] = "Malgun Gothic"
plt.rcParams["axes.unicode_minus"] = False

plt.figure(figsize=(6, 5))
sns.heatmap(
    df[["가격", "판매량", "평점"]].corr(),
    annot=True,          # 숫자 표시
    cmap="coolwarm",     # 색상: 빨강(양) ~ 파랑(음)
    center=0,            # 0을 기준으로 색상 대칭
    vmin=-1, vmax=1,     # 범위 고정
    fmt=".2f"            # 소수점 2자리
)
plt.title("상관관계 히트맵")
plt.tight_layout()
plt.show()
```

> **pandas 버전 참고**: pandas 2.x 이상에서는 `df.corr()`를 쓸 때 비수치 컬럼이 섞여 있으면 경고가 발생할 수 있다. 숫자 컬럼만 미리 선택하거나, `df.corr(numeric_only=True)`처럼 옵션을 주는 습관을 들이면 경고를 피할 수 있다.

**상관관계 분석 시 주의사항:**

1. **상관관계 ≠ 인과관계**: 아이스크림 판매량과 익사 사고 수는 상관관계가 있지만, 아이스크림이 익사를 유발하지 않는다 (공통 원인: 더운 날씨).

2. **비선형 관계는 탐지 못함**: 피어슨 상관계수는 **직선 관계만** 측정한다. U자 형태의 관계는 r≈0으로 나올 수 있다.

3. **이상치의 영향**: 극단값 하나가 상관계수를 크게 바꿀 수 있다. 이상치 처리 후 다시 계산하는 것이 좋다.

4. **다중공선성(Multicollinearity)**: 피처 간 상관관계가 너무 높으면(|r| > 0.9) 둘 중 하나를 제거하는 것이 모델 성능에 유리할 수 있다.

### 5-3a. 실습 — 전처리가 상관계수에 미치는 영향

이 실습에서는 '가격'과 '평점'의 관계를 분석한다. 전처리 전에는 결측치와 이상치 때문에 관계가 흐릿하지만, 전처리 후에는 숨겨진 상관관계가 뚜렷하게 드러나는 과정을 보여준다.

`preprocessing_impact.py` 파일 생성:

```python
import pandas as pd
import numpy as np
import seaborn as sns
import matplotlib.pyplot as plt

# 한글 폰트 설정 (Windows: Malgun Gothic)
plt.rcParams["font.family"] = "Malgun Gothic"
plt.rcParams["axes.unicode_minus"] = False

# ──────────────────────────────────────
# 1. 원본 데이터 생성 (결측치와 극단적 이상치 포함)
# ──────────────────────────────────────
data = {
    "가격": [100, 120, 110, 130, 150, 140, 5000, 10, None, 135],
    #                                      ↑이상치  ↑이상치  ↑결측치
    "평점": [4.0, 4.2, 3.8, 4.3, 4.5, 4.1, 1.0, 5.0, 4.0, None]
    #                                      ↑이상치  ↑이상치       ↑결측치
}
df = pd.DataFrame(data)

# ──────────────────────────────────────
# [Step 1] 전처리 전 상관계수 계산
# ──────────────────────────────────────
# 결측치가 있으면 corr()은 해당 행을 제외하고 계산함
corr_before = df.corr().iloc[0, 1]

# ──────────────────────────────────────
# [Step 2] 데이터 전처리 수행
# ──────────────────────────────────────
df_clean = df.copy()

# 1. 결측치 처리 (중앙값으로 대체)
df_clean["가격"] = df_clean["가격"].fillna(df_clean["가격"].median())
df_clean["평점"] = df_clean["평점"].fillna(df_clean["평점"].median())

# 2. 이상치 처리 (IQR 방식 활용)
def remove_outliers(df, column):
    Q1 = df[column].quantile(0.25)
    Q3 = df[column].quantile(0.75)
    IQR = Q3 - Q1
    lower = Q1 - 1.5 * IQR
    upper = Q3 + 1.5 * IQR
    return df[(df[column] >= lower) & (df[column] <= upper)]

df_clean = remove_outliers(df_clean, "가격")
df_clean = remove_outliers(df_clean, "평점")

# ──────────────────────────────────────
# [Step 3] 전처리 후 상관계수 계산
# ──────────────────────────────────────
corr_after = df_clean.corr().iloc[0, 1]

# ──────────────────────────────────────
# [Step 4] 결과 시각화 비교
# ──────────────────────────────────────
fig, axes = plt.subplots(1, 2, figsize=(12, 5))

# 전처리 전 산점도
sns.scatterplot(x="가격", y="평점", data=df, ax=axes[0], color='red', s=100)
axes[0].set_title(f"전처리 전 (상관계수: {corr_before:.2f})\n*이상치와 결측치로 인해 관계 왜곡")

# 전처리 후 산점도 (회귀선 포함)
sns.regplot(x="가격", y="평점", data=df_clean, ax=axes[1], color='blue', scatter_kws={'s': 100})
axes[1].set_title(f"전처리 후 (상관계수: {corr_after:.2f})\n*정제 후 뚜렷한 양의 상관관계 발견")

plt.tight_layout()
plt.show()

print(f"전처리 전 상관계수: {corr_before:.2f}")
print(f"전처리 후 상관계수: {corr_after:.2f}")
```

**① 상관계수의 드라마틱한 변화**

전처리 전에는 이상치(가격 5000) 한두 개가 무게중심을 완전히 흔들어 놓아, 상관계수가 매우 낮게 (심지어 음수로) 나올 수 있다. 극단값 **하나**가 전체 분석 결과를 뒤집는 것을 눈으로 확인시켜 준다.

**② 의사결정의 차이**

```
전처리 전: "가격과 평점은 관계가 없으니 가격 정책을 마음대로 해도 되겠네?"
           → ❌ 잘못된 결론 (이상치가 만든 허상)

전처리 후: "가격을 올리더라도 품질(평점) 관리가 병행되어야 고객 만족도가 유지되는구나!"
           → ✅ 정확한 결론 (데이터가 말하는 진실)
```

**③ Garbage In, Garbage Out**

아무리 훌륭한 모델을 써도 입력 데이터가 이 상태이면 결과는 쓰레기가 된다. 이것이 CRISP-DM에서 **'데이터 준비' 단계에 전체 시간의 60% 이상**을 투자하는 이유이다.

### 5-4. 데이터 스케일링 (모델 학습 전 필수)

서로 다른 단위와 범위를 가진 변수를 같은 스케일로 맞추는 작업이다. 머신러닝 모델 학습 전에 반드시 수행해야 한다.

**왜 필요한가?**

```
스케일링 전:
  가격:    35,000 ~ 1,200,000  (범위: 1,165,000)
  평점:    3.5 ~ 4.7           (범위: 1.2)

→ 가격이 평점보다 약 97만 배 큰 스케일
→ 모델이 "가격"에만 집중하고 "평점"은 무시하게 됨

스케일링 후 (표준화):
  가격:    -1.2 ~ 1.8          (평균 0, 표준편차 1)
  평점:    -1.5 ~ 1.3          (평균 0, 표준편차 1)

→ 두 변수가 동등한 영향력을 가짐
```

**스케일링이 필요한 모델 vs 불필요한 모델:**

| 스케일링 필요 | 스케일링 불필요 |
|-------------|---------------|
| KNN (거리 기반) | Decision Tree (트리 기반) |
| SVM | Random Forest |
| 로지스틱 회귀 | XGBoost / LightGBM |
| 신경망 (딥러닝) | - |

> 트리 기반 모델은 값의 크기가 아닌 **순서(대소 관계)** 로 분할하므로 스케일링이 필요 없다.

| 방법 | 수식 | 결과 범위 | 특징 |
|------|------|-----------|------|
| **Min-Max 정규화** | (x - min) / (max - min) | 0 ~ 1 | 이상치에 민감, 범위가 고정 |
| **표준화 (Z-score)** | (x - mean) / std | 평균 0, 표준편차 1 | 이상치에 강건, 가장 많이 사용 |

**계산 예시:**

```
데이터: [35000, 89000, 450000, 800000, 1200000]

Min-Max 정규화:
  35000  → (35000 - 35000) / (1200000 - 35000) = 0.00
  89000  → (89000 - 35000) / (1200000 - 35000) = 0.05
  450000 → (450000 - 35000) / (1200000 - 35000) = 0.36
  1200000 → 1.00

표준화 (Z-score):
  mean = 514800, std = 448720
  35000   → (35000 - 514800) / 448720 = -1.07
  89000   → (89000 - 514800) / 448720 = -0.95
  450000  → (450000 - 514800) / 448720 = -0.14
  1200000 → (1200000 - 514800) / 448720 = 1.53
```

```python
from sklearn.preprocessing import MinMaxScaler, StandardScaler

# ── Min-Max 정규화 (0~1) ──
min_max_scaler = MinMaxScaler()
df[["가격_minmax"]] = min_max_scaler.fit_transform(df[["가격"]])

# ── 표준화 (평균 0, 표준편차 1) — 더 자주 사용 ──
standard_scaler = StandardScaler()
df[["가격_standard"]] = standard_scaler.fit_transform(df[["가격"]])

print(df[["상품명", "가격", "가격_minmax", "가격_standard"]].round(2))

# ── 여러 컬럼을 한 번에 스케일링 ──
numeric_cols = ["가격", "판매량", "평점"]
scaler = StandardScaler()
df[numeric_cols] = scaler.fit_transform(df[numeric_cols])
```

**스케일링 시 중요한 규칙 (데이터 누수 방지):**

```python
# ❌ 잘못된 방법: 전체 데이터로 fit한 후 학습/테스트 분리
scaler.fit_transform(df_all)   # 테스트 데이터 정보가 학습에 포함됨

# ✅ 올바른 방법: 학습 데이터로만 fit, 테스트 데이터는 transform만
scaler.fit(X_train)            # 학습 데이터의 평균/표준편차 기억
X_train = scaler.transform(X_train)  # 학습 데이터 변환
X_test = scaler.transform(X_test)    # 테스트 데이터도 같은 기준으로 변환
```

> **fit_transform vs transform**: `fit`은 "기준을 학습"하고, `transform`은 "기준에 맞게 변환"한다. 학습 데이터에만 `fit`을 하고, 테스트 데이터는 학습된 기준으로 `transform`만 해야 한다. 이를 지키지 않으면 **데이터 누수(Data Leakage)** 가 발생하여 모델 성능이 실제보다 좋게 나타나는 오류가 생긴다. (7주차 이후 프로젝트에서 자세히 다룸)

| 상황 | 권장 스케일링 |
|------|-------------|
| 대부분의 경우 | **표준화 (StandardScaler)** |
| 이상치 없고 범위를 0~1로 고정 | Min-Max |
| 이상치가 심한 경우 | **RobustScaler** (중앙값/IQR 기반) |
| 이미지 데이터 (0~255) | Min-Max → 0~1 |

**RobustScaler — 이상치가 극심할 때의 구원투수:**

```python
from sklearn.preprocessing import RobustScaler

# RobustScaler는 평균/표준편차 대신 중앙값/IQR을 사용한다
# → 이상치의 영향을 거의 받지 않음
robust_scaler = RobustScaler()
df[["가격_robust"]] = robust_scaler.fit_transform(df[["가격"]])
```

```
StandardScaler vs RobustScaler 비교:

  데이터: [100, 200, 150, 180, 120, 50000]  ← 50000은 극단 이상치

  StandardScaler: 평균과 표준편차가 이상치에 의해 왜곡됨
    → 정상 데이터들이 -0.2 ~ -0.1 근처에 뭉침 (구분 불가)

  RobustScaler: 중앙값(160)과 IQR(80)로 계산
    → 정상 데이터들이 -0.75 ~ 0.5에 분포 (잘 구분됨)
    → 이상치(50000)만 매우 큰 값으로 튀어나옴

→ 이상치를 제거하기 어려운 경우 RobustScaler가 최선의 선택이다.
```

### 5-4a. 실습 — 스케일링 3종 비교

`scaling_comparison.py` 파일 생성 (경로: `bigdata-project/2w/scaling_comparison.py`):

```python
# ──────────────────────────────────────
# 이상치가 포함된 상품 데이터
```

### 7-2. 메모리 사용량 확인 및 최적화

```python
# 메모리 사용량 확인
print(df.info(memory_usage="deep"))

# 컬럼별 메모리 사용량
mem = df.memory_usage(deep=True) / 1024 / 1024  # MB 단위
print(mem.round(2))
print(f"총 메모리: {mem.sum():.1f} MB")
```

**데이터 타입별 메모리 비교:**

| 타입 | 값 범위 | 메모리 (1만 행 기준) |
|------|---------|---------------------|
| `int64` | -9.2경 ~ 9.2경 | 78 KB |
| `int32` | -21억 ~ 21억 | 39 KB |
| `int16` | -32,768 ~ 32,767 | 19 KB |
| `float64` | 매우 넓음 | 78 KB |
| `float32` | 충분히 넓음 | 39 KB |
| `object` (문자열) | - | 값에 따라 다름 (보통 큼) |
| `category` | - | 고유값이 적을수록 절약 효과 큼 |

**Category 타입의 극적인 메모리 절약 효과:**

```
예: "카테고리" 컬럼 — 고유값 5개 (컴퓨터, 주변기기, 소프트웨어, 네트워크, 보안)

  10만 행 기준:
    object 타입:   약 6.4 MB  (각 행마다 문자열 전체를 저장)
    category 타입: 약 0.1 MB  (고유값 5개만 저장 + 정수 인덱스로 참조)

    → 98% 메모리 절약! (6.4MB → 0.1MB)

  100만 행이면:
    object:   ~64 MB → category: ~0.2 MB  (99.7% 절약)

→ 고유값이 적을수록(성별, 지역, 등급 등) 절약 효과가 극적이다.
→ 반대로 고유값이 많은 컬럼(이름, 주소 등)은 효과가 거의 없다.
```

```python
# 메모리 최적화 함수
def optimize_dtypes(df):
    """데이터프레임의 메모리 사용량을 줄이는 함수"""
    before = df.memory_usage(deep=True).sum() / 1024 / 1024

    for col in df.columns:
        col_type = df[col].dtype

        if col_type == "object":
            # 고유값이 전체의 50% 미만이면 category로 변환
            if df[col].nunique() / len(df) < 0.5:
                df[col] = df[col].astype("category")

        elif col_type in ["int64", "int32"]:
            # 값 범위에 맞는 최소 타입 사용
            if df[col].min() >= 0 and df[col].max() < 65535:
                df[col] = df[col].astype("uint16")
            elif df[col].min() >= -32768 and df[col].max() < 32767:
                df[col] = df[col].astype("int16")
            else:
                df[col] = df[col].astype("int32")

        elif col_type == "float64":
            df[col] = df[col].astype("float32")

    after = df.memory_usage(deep=True).sum() / 1024 / 1024
    print(f"메모리: {before:.1f}MB → {after:.1f}MB ({(1 - after/before) * 100:.0f}% 절약)")
    return df

df = optimize_dtypes(df)
```

### 7-3. 느린 코드 vs 빠른 코드

대용량 데이터에서 성능 차이가 크게 나는 패턴들이다.

#### (1) 반복문 대신 벡터 연산 사용

```python
# ❌ 매우 느림: for 루프 (1만 행 이상에서 체감)
for i in range(len(df)):
    df.loc[i, "할인가"] = df.loc[i, "가격"] * 0.9

# ✅ 빠름: 벡터 연산 (Pandas가 내부적으로 최적화)
df["할인가"] = df["가격"] * 0.9
```

> **핵심 원칙**: Pandas에서 `for` 루프는 최후의 수단이다. 대부분의 작업은 벡터 연산으로 가능하다.

#### (2) apply() 대신 내장 메서드 사용

```python
# ❌ 느림: apply (행마다 Python 함수 호출)
df["가격_로그"] = df["가격"].apply(lambda x: np.log(x) if x > 0 else 0)

# ✅ 빠름: numpy 벡터 연산
import numpy as np
df["가격_로그"] = np.log(df["가격"].clip(lower=1))
```

```python
# ❌ 느림: apply로 조건부 값 할당
df["등급"] = df["평점"].apply(lambda x: "우수" if x >= 4.5 else "보통")

# ✅ 빠름: np.where 또는 np.select 사용
df["등급"] = np.where(df["평점"] >= 4.5, "우수", "보통")

# 조건이 여러 개일 때: np.select
conditions = [
    df["평점"] >= 4.5,
    df["평점"] >= 4.0,
    df["평점"] < 4.0
]
choices = ["우수", "양호", "보통"]
df["등급"] = np.select(conditions, choices)
```

### 7-7. 실습 — 느린 코드 vs 빠른 코드 체감하기

`performance_comparison.py` 파일 생성 (경로: `bigdata-project/2w/performance_comparison.py`):

```python
import pandas as pd
import numpy as np
import time

# ──────────────────────────────────────
# 10만 행 테스트 데이터 생성
# ──────────────────────────────────────
N = 100_000
np.random.seed(42)

df = pd.DataFrame({
    "카테고리": np.random.choice(["컴퓨터", "주변기기", "소프트웨어", "네트워크", "보안"], N),
    "가격": np.random.randint(10000, 2000000, N),
    "판매량": np.random.randint(1, 500, N),
    "평점": np.random.uniform(1.0, 5.0, N).round(1)
})

print(f"테스트 데이터: {N:,}행 × {df.shape[1]}열")
print(f"메모리: {df.memory_usage(deep=True).sum() / 1024 / 1024:.1f} MB")

# ──────────────────────────────────────
# 테스트 1: 새 컬럼 생성 — for 루프 vs 벡터 연산
# ──────────────────────────────────────
print("\n" + "=" * 60)
print("테스트 1: 할인가 계산 (가격 × 0.9)")
print("=" * 60)

# ❌ 느린 방법: for 루프
start = time.time()
for i in range(len(df)):
    df.loc[i, "할인가_slow"] = df.loc[i, "가격"] * 0.9
t_slow = time.time() - start
print(f"  ❌ for 루프:   {t_slow:.3f}초")

# ✅ 빠른 방법: 벡터 연산
start = time.time()
df["할인가_fast"] = df["가격"] * 0.9
t_fast = time.time() - start
print(f"  ✅ 벡터 연산:  {t_fast:.5f}초")
print(f"  → 벡터 연산이 {t_slow / t_fast:.0f}배 빠름!")

# ──────────────────────────────────────
# 테스트 2: 조건부 값 할당 — apply vs np.where
# ──────────────────────────────────────
print("\n" + "=" * 60)
print("테스트 2: 등급 분류 (평점 >= 4.5 → '우수', 아니면 '보통')")
print("=" * 60)

# ❌ 느린 방법: apply + lambda
start = time.time()
df["등급_slow"] = df["평점"].apply(lambda x: "우수" if x >= 4.5 else "보통")
t_slow = time.time() - start
print(f"  ❌ apply:      {t_slow:.3f}초")

# ✅ 빠른 방법: np.where
start = time.time()
df["등급_fast"] = np.where(df["평점"] >= 4.5, "우수", "보통")
t_fast = time.time() - start
print(f"  ✅ np.where:   {t_fast:.5f}초")
print(f"  → np.where가 {t_slow / t_fast:.0f}배 빠름!")

# ──────────────────────────────────────
# 테스트 3: Category 타입 메모리 절약
# ──────────────────────────────────────
print("\n" + "=" * 60)
print("테스트 3: Category 타입 변환 효과")
print("=" * 60)

mem_before = df["카테고리"].memory_usage(deep=True) / 1024
df["카테고리"] = df["카테고리"].astype("category")
mem_after = df["카테고리"].memory_usage(deep=True) / 1024

print(f"  변환 전 (object):   {mem_before:.1f} KB")
print(f"  변환 후 (category): {mem_after:.1f} KB")
print(f"  → {(1 - mem_after / mem_before) * 100:.0f}% 메모리 절약!")

# ──────────────────────────────────────
# 테스트 4: 다운캐스팅 효과
# ──────────────────────────────────────
print("\n" + "=" * 60)
print("테스트 4: 숫자 타입 다운캐스팅")
print("=" * 60)

mem_before = df[["가격", "판매량"]].memory_usage(deep=True).sum() / 1024
df["가격"] = df["가격"].astype("int32")
df["판매량"] = df["판매량"].astype("int16")
mem_after = df[["가격", "판매량"]].memory_usage(deep=True).sum() / 1024

print(f"  변환 전 (int64):    {mem_before:.1f} KB")
print(f"  변환 후 (int32/16): {mem_after:.1f} KB")
print(f"  → {(1 - mem_after / mem_before) * 100:.0f}% 메모리 절약!")

print("\n" + "=" * 60)
print("결론: 10만 행 이상에서는 벡터 연산 + 타입 최적화가 필수!")
print("=" * 60)
```

## 마무리 — AI 시대에 데이터 분석가의 역할

> **Pandas 코드를 외우는 것은 중요하지 않습니다.**
> ChatGPT, Copilot 등 AI가 코드는 대신 작성해줄 수 있습니다.
>
> 하지만 AI가 대신할 수 없는 것이 있습니다:
>
> - **"이 결측치를 평균으로 채워도 되는가?"** 를 판단하는 도메인 지식
> - **"이 이상치가 오류인가, 의미 있는 데이터인가?"** 를 구별하는 경험
> - **"이 데이터에서 어떤 특성을 뽑아야 의미가 있는가?"** 를 발상하는 창의력
> - **"전처리 결과가 비즈니스 맥락에서 타당한가?"** 를 검증하는 비판적 사고
>
> 이번 학기 프로젝트를 통해 여러분이 길러야 할 것은
> **코드를 작성하는 능력**이 아니라 **데이터를 보는 눈**입니다.
>
> 코드는 도구이고, **판단은 사람의 몫**입니다.

**3가지 실습 시나리오:**

| 시나리오 | 함수 | 설명 |
|----------|------|------|

| 3. 결합 후 카테고리별 집계 | merge → groupby | 실전에서 자주 쓰는 "결합 → 집계" 패턴 |

```
left:  A04(판매 없음) NaN으로 유지, A05 제외 → 왼쪽 기준 전부 유지
```

```
  → 순서가 있는 범주에 사용 (불만족 < 보통 < 만족 < 매우만족)

Step 4: 최종 피처 구성
  → 문자열 없는 순수 숫자 데이터 완성 = 모델 입력 준비 완료
```

#### 6. `preprocessing_impact.py` — 전처리 전후 상관계수 비교

**관련 섹션**: 5-3a

**이 실습의 핵심 메시지**: 이상치 **하나**가 전체 분석 결과를 뒤집을 수 있다.

가격과 평점의 관계를 분석할 때, 전처리 전에는 이상치(가격 5000) 때문에 상관관계가 거의 없거나 음수로 나오지만, 결측치 대체 + IQR 이상치 제거 후에는 강한 양의 상관관계(r ≈ 0.85)가 드러난다.

**시각화 출력:**
- 왼쪽: 전처리 전 산점도 (이상치가 관계를 왜곡)
- 오른쪽: 전처리 후 산점도 + 회귀선 (명확한 양의 관계)

> **필수 라이브러리**: `matplotlib`, `seaborn` (사전 설치 필요)

#### 7. `scaling_comparison.py` — 스케일러 3종 비교

**관련 섹션**: 5-4a

같은 데이터에 MinMaxScaler, StandardScaler, RobustScaler를 각각 적용하고, **이상치가 있을 때 어떤 스케일러가 정상 데이터를 가장 잘 구분하는지** 비교한다.

**핵심 발견:**
```
이상치(서버장비: 가격 5000)가 있을 때 정상 데이터(3.5~120)의 스케일링 결과:

  MinMaxScaler:    0.000 ~ 0.023  → 구분 불가 (뭉침)
  StandardScaler: -0.2  ~ -0.1   → 구분 어려움 (뭉침)
  RobustScaler:   넓게 분포       → 구분 가능! ← 이상치에 강건
```

**추가 학습**: train/test 분리 후 올바른 스케일링 순서(fit은 train만, transform은 둘 다)도 포함되어 있다.

> **필수 라이브러리**: `scikit-learn` (사전 설치 필요)

#### 8. `performance_comparison.py` — 느린 코드 vs 빠른 코드

**관련 섹션**: 7-7

**10만 행** 데이터에서 같은 결과를 내는 코드의 성능 차이를 직접 측정한다. "왜 for 루프를 쓰면 안 되는가?"를 숫자로 체감할 수 있다.

**4가지 벤치마크:**

| 테스트 | 느린 방법 | 빠른 방법 | 예상 차이 |
|--------|-----------|-----------|-----------|
| 할인가 계산 | for 루프 | 벡터 연산 (`df["가격"] * 0.9`) | 수백~수천 배 |
| 등급 분류 | apply + lambda | `np.where()` | 수십 배 |
| Category 변환 | object 타입 유지 | `.astype("category")` | 메모리 90%+ 절약 |
| 다운캐스팅 | int64 유지 | int32/int16 변환 | 메모리 50~75% 절약 |

> **주의**: 테스트 1(for 루프)은 10만 행 기준 수십 초가 걸릴 수 있다. 이것이 바로 for 루프를 쓰면 안 되는 이유이다.
